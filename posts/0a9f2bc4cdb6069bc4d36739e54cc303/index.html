BusHound详细使用说明

Bus Hound不同功能界面通过上面一排按钮进行选择

 

一、Capture 数据抓取界面

 

数据抓取界面就是用来显示抓取了数据的界面，这个界面本身需要说明的东西不多，但是其它界面跟数据抓取界面相关的内容比较多，所以介绍的过程中如果有涉及到的，就一并介绍了。

 

1. 数据标签栏

数据标签栏即为上图红框中的内容，用来说明这一列数据的作用。这个标签可以进行设置，可以增加说明项或删减说明项，设置的位置是在Settings界面下方。

说明一下各个选项的内容：

(1) Device:

该栏位显示的是设备和设备端口号(只有USB设备会有端口号)，比如我现在使用的U盘显示

分别对应的是USB的不同端口，可以在Devices界面当中点对应端口可以看到详细信息

选择其中一个，会显示详细信息

 

(2) Address

我也没看懂这是做什么的。实际上在传输过程中这个一直是空的，应该是别的总线会用到的。Bus Hound原文说明如下

(3) Length

用来显示本次传输的数据量，以Byte为单位

 

(4) Phase

基于各类协议，对本次传输数据类型进行一个描述。例如SCSI协议当中，主要分为命令和数据，而数据主要是IN和OUT两种，所以在USB BULK传输当中，这里通常是CMD，OUT，IN三种。还有一种是在上盘时候，使用的是USB Control传输，这时候显示的是CTL

Bus Hound自带的说明当中详细列出了所有Phase类型

 

(5) DATA

传输的数据

 

(6) Descr

如果Phase为OUT或IN的话，将DATA的内容按照ASCII码转换后显示在这里。如果是其他类型，会按照对应协议这次传输进行说明，例如上面的Get Descriptor，这是USB枚举阶段进行的指定操作。比较常见的有

0x25，读容量

0x28，读数据

0x2A，写数据

 

(7) Delta

每一段数据之间间隔的时间，统计的为当前数据传输完成，与上一次数据传输完成的间隔时间。

 

(8) Cmd.Phase.Ofs

三个数据中的第一个表示命令(Cmd)，从1开始统计，每当设备收到一条新的命令就加1。

第二个表示Phase在本次(Cmd)中的位置，每次有一个数据或者状态就加1。

第三个表示数据在这一个Phase当中的位置。

 

最后还有一个括号，里面表示重复的数据被折叠次数，在Settings当中可以设置重复的数据合并显示。

 

(9) Date

日期

 

(10) Time

每一段数据完成时候的时间戳，最大精度为毫秒，小时使用的24小时制

 

(11) Driver

进行这个操作的驱动

 

(12) 复制数据

Capture界面显示的数据可以直接复制，使用Shift选择了一段数据之后，Ctrl+C就可以复制到文本编辑器里，会自动带上BusHound的头

复制过来的数据会跟当前界面换行保持完全一致，所以想一行多一点数据，可以把Bus Hound最大化之后再复制

 

(12) 隐藏功能说明

凡是这一个区域内的配置项，都包含一个隐藏功能。就是即使在之前抓取数据的过程当中没有勾选对应项，Bus Hound依旧会抓取对应的数据进行储存，最后停下来的时候，勾选上想要看的栏目，就能够显示出来。

 

我在抓取数据的时候，没有勾选Delta，那么显然最后看数据的时候是没有Delta这栏的。

 

但是这个时候突然发现“哎呀怎么没抓Delta啊忘了勾上了，这不是白抓了吗阿西..”，这时候需不要惊慌，打开Settings，勾上Delta，间隔时间就全部出现了。

2. 搜索栏

搜索栏可以用来搜索 所有 字符，不管是什么栏位的。所以在进行数据搜索的时候要谨慎，因为如果数据量过大，而准备搜索的字符并不包含在里面，那么Bus Hound就会搜索非常非常非常久。(如果这时候你啪的一下站起来说复制到UE搜索，我只能说这篇是讲BusHound的)

 

在使用搜索栏时，有一个小技巧是搜索指定命令。

输入的越全搜索的越快，这样做就算搜不到，也可以快一点结束搜索。

另外这个搜索据我所知是不支持正则表达式的。

 

 

3. 存储容量进度

 

该进度条显示的是当前Bus Hound抓取界面存储的容量百分比，如果进度条到头了，那么它会清除掉当前所有抓取到的数据，重新开始显示。

能够成功进行上述自动清除操作的前提是，Capture的光标位于最后一行。

Bus Hound可以在抓取的过程中停止数据页面滚动，只需要用滚轮滚一下即可。但是这样存在一个可能导致BusHound崩溃的隐患，因为界面停止在这里，但是数据还在一直抓取。数据存满了缓存之后，理论上Bus Hound应该要刷新，却因为界面停止在这里无法刷新，这时候它就出Bug了，只有关掉重开才能正常显示数据...所以如果要长时间抓取数据，一定要检查一下光标是不是在最后一行，在键盘上按一下 End 键即可。

 

存储的容量可以通过两个参数设置，在Settings界面

 

上面一个Capture Capacity表示的是总存储容量，想要看多一些的话可以不用太约束自己，奔放一点直接来一个9999999。这时候你可能会担心“啊一下输入这么多，出bug或者内存爆炸了怎么办”，这时候不用慌张，输入好之后，随便切个界面再回来一看

Bus Hound已经自动修正这个值在可接受范围内了。(再次鼓掌！)

 

下面一个Max Record Length表示一段数据最大保存容量，比如传输过程中数据经常几十k，我并不想全部抓下来看，可以设置小一点。

像这一段他实际上数据有8k，但是只抓取了前32 Bytes。

 

特别说明，这个功能可不像上面标签栏的功能一样，抓完再改就能显示出来了，这个设置的是抓取时候的数据量。

 

4. Run和Stop

Run，开始抓取数据。

Stop，停止抓取数据。

 

如果点了Stop再点Run，会清掉所有已抓取数据重新抓取。

 

5. 键盘和鼠标指令格式附录

 

Bus Hound自带的说明还列出来鼠标和键盘的中断传输数据格式，感兴趣的可以了解一下

 

二、Save 数据保存界面

 

Save界面非常简单明了，就是将Capture界面所抓取的所有数据保存出来。实际上该保存功能与直接从Capture界面复制数据出来保存的数据是基本一致的，差别是Save只能保存所有数据，而且如果数据量比较大的话Save会快一些。

 

三、Settings 设置界面

该界面大部分功能，在上面介绍Capture 界面的时候已经顺带一起介绍了。这里再补充说明一些上面没介绍到的功能。

 

1. 自动停止数据抓取

 

Bus Hound提供了在触发某些事件的时候自动停止抓取的功能，挨个解释一下触发点：

 

Buffer Full: 当缓存存满了的时候，即存储容量进度满了之后，停止数据抓取

Bus Reset: 总线发生Reset事件的时候，停止数据抓取，Reset在SCSI当中一般是刚通电或者通讯异常的时候会发生

Not Ready: 当设备发送Not Ready状态的时候停止数据抓取。这个状态在存储设备看不到。

Invalid Command: 当发生无效命令事件的时候停止抓取，这个是在Host发送不符合协议或者设备不支持的命令的时候会发生。

No Media: 当Host往指定端口发送命令，但是设备突然断开链接的时候会触发这个事件

Data Overrun: 直接翻译是数据溢出，但是目前还没有发现过类似问题，应该是Host端异常触发的。说明书解释如下：

Media Error: 当设备通讯发生异常，命令回复失败的时候会触发这个事件，与No Media基本类似。

Vendor Error: 枚举阶段的Vendor标识码(Vendor Unique)数据格式不符合规范

Hardware Error: 硬件错误，一般是驱动设备异常

Any Other Error: 其他任何类型错误

 

除此之外，还可以在检测到任何指定数据的时候停止抓取

比如当检测到抓取到0x00数据的时候就停止，那么很快就触发到停止了

 

 

2. Phase to Capture

上图中这些选项是用来选择需要抓取的Phase类型，通常观察传输的话抓取CMD IN OUT三个就完全足够了。如果是掉盘或其他异常之类的，可以再抓取RESET SENSE USTS。

关于SENSE异常的各类状态，在网上找到份比较齐全的表格，放在附录1当中。不过说一下Debug经验之谈，作为Device端进行调试时，一般没必要深究各类Sense的含义，因为基本上都是FW异常掉盘导致的，只有在调试Driver层面的时候会需要确切的异常状态来定位。

 

3. Driver I/O to Capture

这些选项是用来控制显示在Device界面的项目的，一般不需要进行更改，全选即可。

2. 发送命令界面

下面介绍BusHound第二常用且十分好用的发送命令功能，，该功能极其强大，堪称私有命令调试神器，能够发送任何SCSI、ATA甚至是USB命令。

选好要发送的设备后，直接双击要发送的设备，或者点击右下方的Send Commands按钮，即可打开发送命令界面。强调一下，如果更换了设备进行调试，最好重新打开这个发送命令界面，因为不同设备上盘的时候有可能出现端口不一样的情况，而这个界面是根据端口来发送命令的。

(1) 发送命令基本操作

USB和ATA我没怎么用过，所以这里就单独说一下SCSI命令发送作为例子吧。点击选项卡第一个SCSI/ATAPI，就进入准备发送SCSI命令的界面了

上面一排按钮，Run就是发送命令，Reset Device就是Reset Device(废话)，其他不说了，Load是指可以通过文件来加载要发送的数据到数据窗，数据窗也是可以直接修改的。如果是发送数据，则Bus Hound会把数据窗当中的数据发送出去。如果是接收数据，数据窗会用来显示获取到的数据。

 

Command是可以选择SCSI规定好的一些命令，实际上就相当于他给了一份OpCode列表，另外会自动帮忙选一下数据的传输方向，其他啥都没有了。

比如写了28之后，数据方向会自动选到Data In。

 

之后就可以按照CBW的结构体格式来输入数据，这里做个比较简单例子。比如我要往LBA为0x12345678的地址写入0x08扇区数据。

需要注意的是，除了要填好CBW字段当中长度的00 08之外，要在Data Length天上准备传输的数据Byte数量，因为很多SCSI命令是没有在CBW当中标记数据长度的。关于Timeout，是设置SCSI的Timeout时间，如果超过这个时间数据还没有返回的话，会在SCSI层面报Timeout错误。

 

(2) 发送命令异常

再说明一下发送命令的时候比较常见的一些异常：

invalid command

常见的有三种情况

1：Data Length与CBW字段当中长度不匹配。

2：设备掉盘，或者没有设备。非常常见的情况是，低头一看我设备接在电脑上呐！盘符也在怎么会是掉盘。这时候冷静一下，仔细回想一下，这个Send Command窗口是不是操作别的设备的时候打开的...这个窗口只能对指定端口进行操作，而不是一类设备。USB连续上盘多个设备，端口通常都是不一样的，端口也并不代表你插线时候看到的接口，不同接口端口可能是一样的，反之亦然。

3：Device不支持该命令。

 

另外还有一个要注意的是，在发送命令异常之后，该界面的Data length会清零，下次发送前一定要记得填上去。
